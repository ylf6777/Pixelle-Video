Step 1 · 定义面板地址
BASE_URL 即当前控制面板地址（与本文档页的 origin 一致）。示例中会替换为实际地址。
Step 2 · 健康检查（面板存活）
/api/health 是最轻量的存活探针。返回 { status: &quot;ok&quot;, uptime: 秒数 } 即视为面板可用，常用作监控脚本的就绪探针。
Step 3 · GPU 与无卡模式判断
AutoDL 的「无卡模式」也能开机但跑不了生成。返回里 hasGPU=false 就是无卡模式，需到 AutoDL 控制台正常开机；hasGPU=true 时可在 gpus[] 里看到显卡型号和显存大小。
Step 4 · 查询 ComfyUI 状态
返回 { running, starting, reason, port: 6006, ... }。running=true 直接进入生成流程；running=false 时若 starting=true 说明面板已经在自启动，等几秒再轮询即可；都为 false 才需要主动调用启动接口。
Step 5 · 启动并等待 ComfyUI 就绪
useProxy + proxyType=&quot;network_turbo&quot; 会在启动前 source /etc/network_turbo，解决拉模型/插件依赖的网络问题；普通环境可去掉这两个参数。
启动接口立即返回，需要轮询 status 直到 running=true，建议每 3 秒查一次、最多等 60 秒。

当前 ComfyUI 是否运行 + 启动阶段
推荐直接用两个布尔字段判断三阶段：
• running=true               → 启动成功，可直接 generate
• running=false, starting=true → 启动中（进程刚起 HTTP 还未就绪、或面板自启动窗口期）
• running=false, starting=false → 未启动
reason 字段是调试用的细分原因，可选值：
• &quot;ready&quot;             ：进程在且 ComfyUI /system_stats 已就绪（== running:true）
• &quot;starting&quot;          ：进程已起但 HTTP 未就绪，或面板正在自启动等待期
• &quot;process_not_found&quot; ：进程不存在，且没有任何启动流程在跑
建议 generate 前每 2~5 秒轮询一次直到 running=true。

Step 1 · 面板地址与 AutoDL Token
BASE_URL 是当前调度接口服务地址；TOKEN 在 AutoDL 控制台申请。用户自己的集群程序可以把它当作调度 API 地址，不需要依赖面板内置的资产管理流程。
Step 2 · 列出实例并挑选镜像机
返回 { list: [{ instance_uuid, status, machine_alias, ... }] }。
status 可能是 running / shutdown / starting。这里挑两台 shutdown 的当镜像机。
Step 3 · 开机（有卡模式）
默认 power_on 是有卡模式（跑生成必须有卡）。省钱场景可加 mode: &quot;shutdown&quot; 走无卡模式，但无卡模式下跑不了生成。
Step 4 · 等待开机并解析镜像机面板 URL
每台镜像机内部也运行着一份面板服务（端口 6008），AutoDL 给它一个对外域名 snapshot.service_6008_domain，前面拼 https:// 就是该镜像机的 mirror_url。
开机需要时间，所以轮询 snapshot 接口直到出现这个字段（5 秒查一次）。
Step 5 · 在各镜像机上启动 ComfyUI
该接口非阻塞，命令发出后镜像机会异步启动 ComfyUI；要确认是否启动完毕，需要轮询 /api/concurrent/mirror/comfy-status。
Step 6 · 提交异步批跑任务
这是并发接口的核心：调度器会在 mirrors/targets 里寻找空闲机器，空闲机器立即接收下一个 job；如果多台机器都在跑，剩余 job 会留在队列里等待，等任意机器完成当前任务后继续派发。这样用户程序只需要提交一批 jobs，不需要自己判断哪台机器空闲。
• workflow_id：要跑的工作流（注意：每台镜像机里都要先有这个工作流文件，否则得先用 /api/workflow/files/update 同步过去）
• jobs：每个 job 是一次独立生成；id 任意填，用来在结果里对回去
• targets：可选，不传则用全部可用镜像机
返回 { run_id }，是这次批跑的 ID，用它查进度。
Step 7 · 轮询进度并自行下载结果
每隔 2 秒查一次 run-batch-status：返回 { status, done, total, error, results: [...] }。
当 status 是 completed/cancelled/error 时，遍历 results：每个完成的 job 都有 target_url（哪台镜像机跑的）和 prompt_id（结果 ID）。用户自己的程序应使用 target_url + prompt_id 去对应镜像机查询结果，再把文件下载到自己的项目目录、数据库或对象存储中。

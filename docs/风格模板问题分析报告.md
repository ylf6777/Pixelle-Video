# 风格模板显示问题 — 根因分析与修复报告

## 一、问题现象

用户访问 `/styles` 页面后，每个风格模板卡片的缩略图区域无法显示内容，仅显示渐变色占位块或空白。

## 二、排查过程与根因

| 轮次 | 尝试方案 | 结果 | 根因 |
|------|---------|------|------|
| 1 | 使用 `<iframe>` 加载模板 HTML | 不显示 | **CSP 策略** `frame-ancestors 'none'` 阻止了所有 iframe 嵌入。修复：改为 `'self'` |
| 2 | 修复 CSP 后仍用 iframe | 加载慢/空白 | **25 个 iframe 同时加载**，每个 1080×1920 模板 ~10KB，浏览器阻塞 |
| 3 | 改用 `srcdoc` 属性预注入 HTML | 不显示 | Jinja2 `|e` 转义破坏了 srcdoc 的 HTML 结构，且页面体积膨胀到 ~200KB |
| 4 | 彻底移除 iframe/srcdoc，改用渐变占位 | 占位正常但无真实预览 | 违背需求——用户需要看到模板效果，不是色块 |
| 5 | 直接打开模板 HTML 链接 | 显示 `{{title}}` 原始占位符 | **模板 HTML 包含 Jinja2 变量**（`{{title}}`, `{{narration}}` 等），静态文件服务未处理 |
| 6 | 服务端替换占位符后再返回 | 文字正常但图片缺失 | 模板中的 `{{image}}` 占位符替换为空字符串，无实际图片 |

### 最终根因

**原网站在 `docs/images/` 目录下预先生成了 48 张模板截图（JPG/PNG）**，通过 `style_config.py` 中的 `get_template_preview_path()` 函数加载这些截图作为卡片缩略图。新 Web UI 从头开发时未迁移这些现有资源，一直在试图用 iframe/srcdoc 等方案实时渲染模板——这完全绕过了原网站已有的预览图片资产。

## 三、最终修复方案

1. **回归原方案**：直接使用 `docs/images/1080x1920/{模板名}.jpg` 的预生成截图
2. **路由加载**：`/styles` 页面自动扫描 `docs/images/` 目录匹配每个模板的预览图
3. **CSS 适配**：风格卡片缩略图使用 `object-fit: contain` + `aspect-ratio: auto` 完整显示 9:16 竖版截图
4. **静态文件挂载**：`app.mount("/docs/images", ...)` 使图片可通过 HTTP 访问

## 四、最终状态

- 25 个模板卡片全部显示真实截图
- 图片完整无裁切、无变形
- 页面加载仅需加载图片（无 iframe 开销）
- 与原始 Streamlit 网站的预览效果完全一致

## 五、教训

**新功能开发前应先完整阅读原代码的资产和数据流，不要重新发明轮子。** 原网站投入了大量工作预生成模板截图，这些资产直接可用，却被忽略了。
